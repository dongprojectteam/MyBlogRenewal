# [요구사항 명세서] 수도쿠 (Sudoku) - Next.js App

## 1. 프로젝트 개요

* **게임 제목:** 수도쿠 (Sudoku)
* **플랫폼:** Next.js 기반 블로그 내 임베디드 웹 앱 (`app/tetris`와 동일한 페이지 셸·헤더·비주얼 톤)
* **핵심 루프:** 레벨(난이도)을 고른 뒤, 자동 생성된 9×9 퍼즐을 규칙에 맞게 완성한다. 완료 시 **클리어까지 걸린 시간**이 해당 레벨의 기록(점수)이 되며, Supabase에 저장되어 **글로벌 리더보드**에서 조회된다.
* **렌더링:** 게임 보드·강조·숫자 패드 등 **주 UI는 HTML5 Canvas**로 그린다. (레이아웃 래퍼·탭·폼 등은 React/DOM으로 두되, `tetris`와 이질감 없는 타이포·간격·글래스 패널 스타일을 유지한다.)

## 2. 레벨 설계 (약 10단계)

* **레벨 개수:** 기본 **10개** (`Level 1` ~ `Level 10`). 숫자는 1부터 시작하는 정수 ID로 관리한다.
* **난이도 진행:** 레벨이 올라갈수록 퍼즐 생성 파라미터(시도 횟수, 제거 목표, 솔버가 요구하는 기법 등)가 **엄격해지도록** 설계한다. 구체적인 임계값은 **별도로 제공되는 생성 알고리즘**이 결정한다.
* **레벨 선택 UI:** `app/tetris`의 모드 탭(`.tetris-mode-tabs`)과 유사한 **레벨 탭/칩**으로 선택한다. 각 탭에 짧은 부제(예: “입문”, “중급” …)를 둘 수 있다.
* **세션 단위:** 한 세션에서는 **선택한 레벨 하나**에 대해 하나의 퍼즐이 생성·플레이된다. 레벨 변경 시 새 퍼즐 생성 플로우로 진입한다.

## 3. 퍼즐 생성 및 알고리즘 연동

구현 시 **생성 로직은 별도 모듈**(또는 웹 워커)로 분리하고, UI는 아래 **계약**만 만족하면 된다.

### 3.1 생성 파이프라인 (제공 예정 알고리즘과의 정합)

다음 단계를 문서화된 순서로 수행하는 것을 전제로 한다.

1. **정답판 생성:** 백트래킹 등으로 완전한 보드를 채우고, 셔플 등으로 **매판 다른 결과**가 나오게 한다.
2. **숫자 제거 및 유일성:** 완성판에서 칸을 제거할 때마다 **해가 유일한지**(솔버로 `countSolutions` 또는 동등 검증) 확인한다. 유일해가 깨지면 되돌린다.
3. **난이도 판정:** 빈칸 개수만이 아니라, 솔버가 사용한 **논리 기법(예: Sole Candidate → Naked Pairs → X-Wing …)** 에 따라 난이도를 확정한다.

성능 메모(요구사항): 제거 시 **대칭 제거** 등으로 시도 횟수를 줄이는 최적화를 알고리즘 측에서 허용한다. 고난이도에서 유일해 유지 실패가 잦으면 **시도 횟수 상한** 후 정답판을 폐기하고 **처음부터 재생성**하는 정책을 둔다.

### 3.2 UI ↔ 생성기 인터페이스 (제안)

* **입력:** `levelId` (1–10), 선택적 `seed` (재현/디버그용).
* **출력:**
  * `puzzle`: 9×9, 고정값(힌트)은 1–9, 빈 칸은 `0` (또는 `null` — 구현 시 하나로 통일).
  * `solution`: 9×9 정답 (클리어 검증·힌트 금지 칸 판별에 사용).
  * `metadata`: (선택) 생성에 걸린 시간, 적용된 최고 난이도 기법 라벨, 시드 등.
* **비동기:** 생성은 `async` 또는 Worker 메시지로 수행하며, 메인 스레드에서 **프로그레스 콜백**(`0.0` ~ `1.0` 또는 단계 라벨)을 받을 수 있어야 한다.

### 3.3 생성 중 UX (프로그레스 바)

* 생성이 **즉시 끝나지 않는 경우**(레벨이 높거나 유일해 검증 비용이 큰 경우) 화면에 **진행률 표시**를 둔다.
  * Canvas 위 오버레이 또는 패널 내 **프로그레스 바 + 짧은 상태 문구**(예: “유일해 검증 중…”)를 사용한다.
  * `tetris`의 로딩/오버레이 톤과 맞출 것.
* 생성 중에는 **시작/입력 비활성화** 또는 취소 버튼 정책을 명시한다. (기본: 취소 시 이전 화면 또는 레벨 선택으로 복귀.)

## 4. 게임플레이 규칙

* **표준 스도쿠:** 각 행·열·3×3 박스에 1–9가 한 번씩.
* **고정 칸:** 초기 퍼즐에서 비어 있지 않은 칸은 **편집 불가**이며 시각적으로 구분한다.
* **플레이어 입력:** 빈 칸에만 1–9 입력 가능. 잘못된 숫자(규칙 위반)에 대한 피드백은 아래 **검증 정책**에 따른다.
* **승리 조건:** 모든 빈 칸이 채워지고 규칙을 만족하면 클리어. 타이머 정지 후 결과 패널 표시.
* **일시정지(선택):** `tetris`와 같이 일시정지 시 타이머 정지 여부를 정의한다. (권장: 일시정지 시 시간 정지.)

### 4.1 검증·피드백 정책 (구현 결정용)

요구사항 수준에서 다음 중 하나를 **design 단계에서 확정**한다.

* **엄격:** 규칙 위반 입력을 **거부**하거나 즉시 표시(빨간 표시)만 허용.
* **관대:** 위반 숫자 입력은 허용하되 충돌 셀 하이라이트(구현 선택).

기본 권장: **충돌 하이라이트 + 최종 클리어는 오직 유효한 완전판만 인정.**

## 5. 입력 장치

### 5.1 키보드

* **숫자:** `1`–`9`로 선택 셀에 입력. `Backspace`/`Delete`로 지우기.
* **이동:** 화살표 또는 `WASD`로 선택 셀 이동(둘 중 하나 또는 병행 — `design`에서 확정).
* **기타:** `tetris`와 충돌하지 않는 범위에서 일시정지 키 등을 정의할 수 있다.

### 5.2 마우스·터치

* **셀 클릭:** 선택 포커스 이동.
* **숫자 패드:** Canvas 상(또는 보드 옆 패널을 Canvas로 그릴 경우 동일 스타일)에 **1–9 버튼**과 **지우기**를 두어 한 손 조작이 가능하게 한다.
* **접근성:** 포커스·역할은 가능한 한 DOM 보조 레이블 또는 `aria`로 보완한다. (Canvas만으로는 제한적이므로 숨김 라이브 리전 또는 병행 패턴 검토.)

## 6. 타이머 및 점수(기록)

* **측정값:** 레벨별 **클리어 시간** `timeMs` (밀리초). 짧을수록 좋은 기록이다.
* **시작 시점:** 퍼즐 생성이 완료되고 플레이어가 **Start**를 누른 시점부터 측정하거나, 생성 완료 직후 자동 시작 중 하나를 택한다. (`tetris`는 Start 이후 시작 — **동일하게 Start 이후 권장.**)
* **저장 조건:** 유효한 클리어만 서버 제출. 중복 제출·치트 방지는 API/서버 규칙으로 최소한 검증(예: `solution` 해시 또는 서버 측 재검증은 `design`에서 결정).

## 7. Supabase · 글로벌 리더보드

* **저장소:** 기존 블로그와 동일하게 **Supabase**를 사용한다. 테트리스와 **패턴 정합**: `app/api/.../scores` Route Handler + `lib/data` 유사 함수.
* **조회:** `levelId`별 상위 N건(예: 50) 목록. 정렬: `time_ms` **오름차순**(동일 시간은 `created_at` 등으로 타이브레이크).
* **저장 필드(제안):**
  * `player_name` (문자열)
  * `level` 또는 `level_id` (1–10)
  * `time_ms` (클리어 시간)
  * `puzzle_id` / `seed` / `puzzle_hash` (재현·감사용, 선택)
  * `created_at`
* **클라이언트:** `tetris`와 같이 플레이어 이름(`localStorage` 키 네이밍은 프로젝트 규칙에 맞춤, 예: `dopt-sudoku-player-name`) 및 저장/에러 메시지 UX를 통일한다.

## 8. UI/UX (`app/tetris` 정렬)

* **페이지 구조:** `page-shell` + `SiteHeader` + 클라이언트 게임 컴포넌트.
* **레이아웃:** `.tetris-layout` / `.tetris-play-panel` / `.tetris-side-panel` 과 **동일한 그리드·패널 철학**을 따른다. 클래스명은 `sudoku-` 접두사를 신설하되 **시각적 토큰(간격, 글래스, 탭 스타일)은 tetris와 공유 또는 복제**한다.
* **주요 영역:**
  * 상단: 제목, **레벨 탭**, 현재 레벨 설명.
  * 중앙: **Canvas** 보드(선택 셀, 고정 칸, 충돌/오류 하이라이트, 완성 연출).
  * 측면 또는 하단: 타이머, **New game / Start / Pause**, 로컬 베스트, **글로벌 리더보드** 목록.
* **테마:** 라이트/다크 모드 모두에서 대비 확보. Canvas 내 색은 CSS 변수(`globals.css`의 테마 변수)를 읽어 **tetris·블로그와 톤 일치**시킨다.
* **오버레이:** 준비/일시정지/클리어/생성 중 상태는 `tetris-board-overlay`와 유사한 **반투명 오버레이 카피**로 통일한다.

## 9. 기술 스택 및 품질

* **프레임워크:** Next.js App Router, `"use client"` 게임 컴포넌트.
* **Canvas:** `devicePixelRatio` 대응, 리사이즈(`ResizeObserver`) 시 보드 재배치.
* **상태:** React state + ref; 생성 로직이 무거우면 **Web Worker** 도입을 허용한다.
* **SEO/메타:** `app/tetris/page.tsx` 수준의 `metadata`, OG 이미지(추후 `sudoku-preview.svg` 등), 구조화 데이터는 구현 단계에서 `dopt-seo-indexing` 스킬에 맞춰 반영한다.

## 10. 비범위(이번 요구사항에서 제외)

* 멀티플레이 실시간 대전.
* 힌트 자동 제공(추후 확장 가능).
* 사용자 정의 크기(6×6 등) — **9×9만** 전제.
