# 요구사항 명세서: Sudoku

최종 업데이트: 2026-05-14

이 문서는 현재 구현된 `/sudoku` 코드를 기준으로 스도쿠 게임의 요구사항을 정리한다. 구현 파일은 `app/sudoku`, `lib/sudoku`, `app/api/sudoku/scores`, `public/workers/sudoku-generator.worker.js`, `supabase/sudoku_scores.sql`을 기준으로 한다.

## 1. 목표

- 9x9 표준 스도쿠를 브라우저에서 바로 플레이할 수 있어야 한다.
- Lv 1-10 난이도별로 자동 생성된 퍼즐을 제공해야 한다.
- Canvas 기반 보드와 숫자 패드, 키보드 입력을 모두 지원해야 한다.
- 클리어 결과는 점수, 시간, 실수 수, 레벨, 시드를 기준으로 로컬 베스트와 글로벌 리더보드에 반영해야 한다.
- 서버는 제출된 퍼즐과 정답을 재검증해 조작 가능성을 낮춰야 한다.

## 2. 현재 범위

### 포함

- `/sudoku` 페이지와 SEO 메타데이터
- 난이도 탭 Lv 1-10
- Web Worker 기반 퍼즐 생성
- Canvas 보드, 선택 셀 표시, 숫자 패드, 지우기
- 키보드 이동 및 숫자 입력
- 레벨별 도움 표시 정책
- 점수 계산, 패널티, 로컬 베스트
- Supabase 기반 글로벌 리더보드 및 환경변수 미설정 시 seed 데이터 폴백
- 완료 후 점수 제출 및 서버 검증

### 제외 또는 다음 단계

- 노트 기능, 즉 후보 숫자 메모 기능은 요구사항과 설계를 정의하고 다음 구현 범위로 둔다.
- 일시정지
- 멀티플레이
- 사용자 지정 보드 크기
- 힌트 자동 제공
- 생성 난이도의 풀이 기법 기반 판정

## 3. 주요 사용자 흐름

1. 사용자는 `/sudoku`에 진입한다.
2. 기본 Lv 1 퍼즐이 자동 생성된다.
3. 사용자는 Lv 1-10 탭으로 난이도를 바꿀 수 있다. 난이도 변경 시 새 퍼즐이 생성된다.
4. 퍼즐 생성이 끝나면 `Start` 버튼으로 플레이를 시작한다.
5. 사용자는 보드 셀을 선택하고 숫자 패드, 키보드 숫자, Backspace/Delete로 값을 입력하거나 지운다.
6. 모든 칸이 채워지고 충돌이 없으며 정답과 일치하면 게임이 완료된다.
7. 완료 화면에서 이름을 입력해 점수를 저장한다.
8. 같은 레벨의 글로벌 리더보드와 로컬 베스트가 갱신된다.

## 4. 기능 요구사항

### FR-01 페이지

- `/sudoku` 경로는 `SudokuClient`를 렌더링해야 한다.
- 페이지는 title, description, canonical, OG 이미지, JSON-LD를 제공해야 한다.
- 홈/유틸리티 목록에서 스도쿠 유틸리티로 진입할 수 있어야 한다.

### FR-02 난이도

- 난이도는 `SudokuLevelId = 1 | ... | 10`으로 관리한다.
- 각 레벨은 생성 프로필을 가진다.

| 레벨 | 빈칸 목표 | 제거 시도 | 재생성 라운드 | 사용자 체감 |
| --- | ---: | ---: | ---: | --- |
| 1 | 24 | 120 | 10 | 입문 |
| 2 | 28 | 140 | 11 | 쉬움 |
| 3 | 32 | 160 | 12 | 안정 |
| 4 | 36 | 185 | 13 | 표준 |
| 5 | 40 | 210 | 14 | 중간 도전 |
| 6 | 43 | 235 | 15 | 중급 |
| 7 | 46 | 260 | 16 | 집중 |
| 8 | 49 | 290 | 18 | 상급 |
| 9 | 51 | 320 | 20 | 어려움 |
| 10 | 54 | 360 | 24 | 마스터 |

### FR-03 퍼즐 생성

- 퍼즐 생성은 메인 스레드를 막지 않도록 Web Worker에서 실행해야 한다.
- 생성기는 시드 기반 난수, 백트래킹 완성판 생성, 무작위 칸 제거, 해답 유일성 검사를 사용한다.
- 결과는 `puzzle`, `seed`, `metadata.givenCount`, `metadata.durationMs`를 포함한다.
- 클라이언트는 진행률과 상태 문구를 표시해야 한다.
- 사용자가 생성을 취소하거나 다른 레벨을 선택하면 이전 Worker 결과는 무시되거나 종료되어야 한다.

### FR-04 게임 상태

현재 구현 상태는 다음 5단계다.

- `idle`: 퍼즐 없음 또는 생성 취소 상태
- `generating`: 퍼즐 생성 중
- `ready`: 퍼즐 생성 완료, 시작 전
- `playing`: 타이머 진행 및 입력 가능
- `completed`: 클리어 완료, 제출 가능

일시정지 상태는 현재 요구사항에서 제외하며 향후 확장으로 둔다.

### FR-05 입력

- 마우스/터치로 Canvas 보드 셀을 선택할 수 있어야 한다.
- Canvas 안의 숫자 패드에서 1-9를 입력하고 지우기 버튼으로 셀을 비울 수 있어야 한다.
- 키보드 `Arrow` 또는 `WASD`로 선택 셀을 이동할 수 있어야 한다.
- 키보드 `1-9`, `Numpad1-9`로 숫자를 입력할 수 있어야 한다.
- `Backspace` 또는 `Delete`로 입력 셀을 지울 수 있어야 한다.
- 고정 칸은 변경할 수 없어야 한다.
- 입력은 `playing` 상태에서만 실제 보드 값을 변경해야 한다.

### FR-06 도움과 패널티

레벨별 도움 정책은 `getSudokuAssistProfile`을 기준으로 한다.

| 레벨 | 충돌 표시 | 패널티 표시 | 행/열/박스 강조 | 같은 숫자 강조 | 라벨 |
| --- | --- | --- | --- | --- | --- |
| 1-2 | 모든 충돌 | 항상 표시 | 표시 | 표시 | 전체 도움 |
| 3 | 모든 충돌 | 항상 표시 | 숨김 | 숨김 | 충돌 표시 |
| 4 | 선택 칸 충돌 | 항상 표시 | 숨김 | 숨김 | 선택 칸 충돌 |
| 5 | 짧은 경고 | 항상 표시 | 숨김 | 숨김 | 즉시 경고 |
| 6-7 | 숨김 | 항상 표시 | 숨김 | 숨김 | 실수만 표시 |
| 8 | 숨김 | 약하게 표시 | 숨김 | 숨김 | 실수 약하게 표시 |
| 9-10 | 숨김 | 완료 후 표시 | 숨김 | 숨김 | 완료 후 공개 |

- 실수는 충돌이 발생한 `row:col:value` 조합을 처음 입력했을 때 1회 증가한다.
- 실수 수는 점수 산정의 패널티로 사용된다.
- 레벨/도움 라벨은 보드 내부를 가리지 않도록 보드 밖 플레이 헤더에 표시해야 한다.

### FR-07 점수와 베스트

- 점수는 시간, 레벨, 빈칸 수, 실수 수로 계산한다.
- 계산식은 현재 코드의 `computeSudokuScoreBreakdown`을 기준으로 한다.

```text
baseScore = 10000 + levelId * 1500 + emptyCells * 250
timePenalty = ceil(timeMs / 1000) * penaltyPerSecond
mistakePenalty = mistakeCount * 300
finalScore = max(100, round(baseScore - timePenalty - mistakePenalty))
```

- 초당 시간 패널티는 Lv 1-3: 12, Lv 4-7: 18, Lv 8-10: 25다.
- 로컬 베스트는 더 높은 점수를 우선하고, 점수가 같으면 더 짧은 시간을 우선한다.
- 로컬 베스트 저장 키는 `dopt-sudoku-best-v1`이다.

### FR-08 글로벌 리더보드

- `GET /api/sudoku/scores?level={1..10}`으로 레벨별 기록을 조회한다.
- `POST /api/sudoku/scores`로 완료 기록을 저장한다.
- 리더보드는 점수 내림차순으로 보여준다.
- Supabase 환경변수가 없으면 seed 데이터 기반 폴백 결과를 제공하고 저장은 `saved: false`로 응답한다.
- 플레이어 이름은 2-18자로 제한한다.
- 플레이어 이름 저장 키는 `dopt-sudoku-player-name`이다.

### FR-09 서버 검증

점수 제출 시 서버는 다음을 검증해야 한다.

- `levelId`가 1-10인지 확인한다.
- `timeMs`가 0 이상 1시간 이하인지 확인한다.
- `mistakeCount`가 0 이상 999 이하인지 확인한다.
- `seed`가 허용 범위의 정수인지 확인한다.
- `puzzle`, `playerGrid`, `givenMask`가 81자 형식인지 확인한다.
- 서버가 같은 `levelId`, `seed`, 생성 프로필로 퍼즐을 재생성한다.
- 제출된 `puzzle`과 `givenMask`가 서버 재생성 결과와 일치해야 한다.
- 고정 칸은 제출 중 변경될 수 없어야 한다.
- 완성 보드는 행, 열, 3x3 박스에 충돌이 없어야 한다.
- 완성 보드는 서버 재생성 정답과 일치해야 한다.

### FR-10 UI와 반응형

- 보드와 숫자 패드는 Canvas에 렌더링한다.
- 상단 HUD는 Score, Time, Penalty, Best를 표시한다.
- 레벨/도움 라벨은 보드 위 오버레이가 아니라 플레이 헤더에 표시한다.
- 완료 시 보드 위에 결과 카드와 제출 폼을 표시한다.
- 사이드 패널은 세션 정보, 글로벌 리더보드, 조작 또는 완료 요약을 제공한다.
- 모바일에서는 HUD와 사이드 패널이 세로로 안정적으로 배치되어야 한다.

### FR-11 접근성

- Canvas에는 `aria-label`을 제공한다.
- 현재 선택 셀과 값은 `aria-live` 영역으로 보조 기술에 전달한다.
- 버튼과 입력 필드는 DOM 요소로 유지한다.
- 숫자, 점수, 시간 텍스트는 작은 화면에서도 잘리지 않도록 해야 한다.

## 5. 추가 요구사항: 노트 기능

사용자 피드백으로 받은 노트 기능은 아직 구현되어 있지 않다. 이 기능은 스도쿠에서 흔히 쓰는 후보 숫자 메모, 즉 pencil marks/candidate notes를 의미하며 다음 구현 범위로 정의한다.

### FR-12 노트 모드

- 사용자는 플레이 중 노트 모드를 켜고 끌 수 있어야 한다.
- 노트 모드는 보드 밖 조작 영역에 버튼 또는 연필 아이콘 토글로 제공한다.
- 노트 모드의 현재 상태는 시각적으로 명확해야 하며, 키보드 사용자도 확인할 수 있어야 한다.
- 노트 모드는 `playing` 상태에서만 숫자 입력 동작을 바꾼다.
- `ready`, `generating`, `completed` 상태에서는 노트 입력을 허용하지 않는다.

### FR-13 후보 숫자 입력

- 노트 모드가 켜진 상태에서 빈칸을 선택하고 숫자 1-9를 입력하면 해당 숫자를 후보로 추가한다.
- 이미 같은 후보 숫자가 있으면 다시 입력했을 때 제거한다.
- 후보 숫자는 정답 입력이 아니며 `playerGrid` 값을 변경하지 않는다.
- 후보 숫자 입력은 실수 수, 패널티, 점수에 영향을 주지 않는다.
- 고정 칸에는 노트를 추가할 수 없다.
- 이미 확정 숫자가 입력된 칸에는 노트를 추가할 수 없다.

### FR-14 일반 입력과 노트의 관계

- 일반 입력 모드에서 확정 숫자를 입력하면 해당 칸의 기존 노트는 모두 삭제된다.
- 일반 입력 모드에서 확정 숫자를 지워 빈칸이 되더라도 이전 노트는 자동 복원하지 않는다.
- 지우기 동작은 모드별로 다르게 동작한다.
- 일반 입력 모드의 지우기는 현재처럼 확정 숫자를 지운다.
- 노트 모드의 지우기는 선택 칸의 노트만 모두 지운다.
- 노트 모드에서 노트가 없는 칸에 지우기를 눌러도 확정 숫자를 지우지 않는다.

### FR-15 자동 노트 정리

- 1차 구현에서는 자동 노트 정리를 기본 비활성화한다.
- 같은 행, 열, 3x3 박스에 확정 숫자가 입력될 때 같은 후보를 자동 삭제하는 기능은 별도 옵션으로 확장할 수 있다.
- 자동 정리를 추가할 경우 기본 동작, 토글 위치, 기존 노트 손실 가능성을 별도 UX 검토 후 결정한다.

### FR-16 노트 표시

- 후보 숫자는 셀 내부에 3x3 미니 그리드 형태로 작게 표시한다.
- 후보 숫자는 1-9의 상대 위치가 고정되어야 한다.
- 후보 숫자는 확정 숫자보다 작고 약한 톤이어야 한다.
- 선택 칸, 같은 숫자 강조, 충돌 강조와 함께 표시될 때도 숫자 판독성을 해치지 않아야 한다.
- 모바일 보드에서도 후보 숫자가 너무 작아 읽을 수 없는 상태가 되면 안 된다.
- 보드 위에 별도 노트 설명 텍스트를 올려 숫자를 가리지 않아야 한다.

### FR-17 접근성과 입력 단축키

- 노트 모드 토글은 `aria-pressed`를 제공해야 한다.
- 선택 칸의 후보 숫자 목록은 `aria-live` 문구에 포함해야 한다.
- 키보드 단축키는 기존 입력과 충돌하지 않는 키를 사용한다.
- 권장 단축키는 `N`이며, 입력 폼 안에서는 비활성화한다.
- 숫자 패드와 키보드 숫자는 노트 모드 상태에 따라 동일하게 후보 토글로 동작해야 한다.

### FR-18 저장과 제출

- 노트 데이터는 점수 제출 API에 포함하지 않는다.
- 노트 데이터는 서버 검증에 사용하지 않는다.
- 새 게임, 레벨 변경, 생성 취소 시 노트는 초기화한다.
- 브라우저 새로고침 후 노트 복원은 1차 구현 범위에서 제외한다.
- 향후 중단 이어하기 기능을 만들 경우 노트는 별도 localStorage 스키마로 저장한다.

### 노트 기능 인수 기준

- 노트 모드를 켠 상태에서 숫자를 누르면 셀의 후보 숫자가 토글된다.
- 노트 모드를 끈 상태에서 숫자를 누르면 현재처럼 확정 숫자가 입력된다.
- 후보 숫자는 실수 수와 점수에 영향을 주지 않는다.
- 확정 숫자를 입력한 칸에서는 후보 숫자가 사라진다.
- 고정 칸과 확정 숫자 칸에는 후보 숫자를 추가할 수 없다.
- 노트 모드 지우기는 노트만 지우고 확정 숫자는 지우지 않는다.
- 새 게임과 레벨 변경 시 모든 노트가 초기화된다.

## 6. 알려진 보완점

- 현재 일부 코드 내 한글 UI 문구가 인코딩 깨짐 형태로 남아 있어 문구 정리가 필요하다.
- Supabase 조회 정렬은 현재 점수 내림차순과 생성일 오름차순 중심이다. 점수 동점 시 `time_ms asc`를 보조 정렬로 추가하는 것이 바람직하다.
- 완료 제출 UI가 완료 오버레이와 사이드 패널에 모두 노출된다. 최종 UX에서는 하나의 주 경로로 정리하는 것이 좋다.
- 일시정지는 아직 구현되지 않았다.
- 노트 기능은 요구사항과 설계만 정의되어 있으며 아직 구현되지 않았다.

## 7. 인수 기준

- Lv 1-10에서 퍼즐 생성, 시작, 입력, 완료까지 끊기지 않아야 한다.
- 보드 내부에는 숫자를 가리는 HUD 또는 배지가 없어야 한다.
- 고정 칸은 포인터와 키보드 입력으로 수정되지 않아야 한다.
- 충돌 입력은 레벨별 도움 정책에 맞게 표시되고 실수 수가 증가해야 한다.
- 완료된 보드는 서버 검증을 통과해야만 리더보드에 저장되어야 한다.
- `npm run build`가 성공해야 한다.
