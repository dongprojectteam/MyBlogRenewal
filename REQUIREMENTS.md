# MyBlogRenewal 요구사항

## 1. 프로젝트 개요

이 프로젝트는 `Next.js + TypeScript` 기반으로 제작하는 개인용 블로그형 유틸 허브이다.  
배포는 `Vercel`, 데이터베이스 및 파일 저장은 `Supabase`를 사용한다.

사이트는 처음에는 매우 단순한 형태로 시작하며, 이후 `/week-calendar`, `/diff` 같은 유틸을 서브 경로 형태로 하나씩 추가해 나간다.
또한 나를 소개할 수 있는 공개 소개 페이지도 포함한다.

## 2. 기술 스택

- 프레임워크: `Next.js`
- 언어: `TypeScript`
- 배포: `Vercel`
- 백엔드 / 데이터베이스 / 스토리지: `Supabase`

## 3. 프로젝트 목표

- 홈 화면을 내가 만든 유틸들의 진입 허브로 사용한다
- 나를 소개하는 공개 프로필 페이지를 제공한다
- `/admin` 경로에 관리자 전용 관리 페이지를 제공한다
- 관리자는 메모, 파일, 유틸 메타데이터를 직접 관리할 수 있어야 한다
- 관리자는 내 소개, 약력, 사진, 프로젝트 목록, 외부 링크도 직접 관리할 수 있어야 한다
- 초기 버전은 무료 플랜 범위 안에서 가볍게 운영 가능한 구조를 목표로 한다

## 4. 주요 경로

### 공개 페이지

- `/`
  - 홈 화면
  - 등록된 유틸 목록 또는 빈 상태 화면을 표시
- `/about`
  - 공개 소개 페이지
  - 내 사진, 인사말, 프로젝트 목록, 외부 링크를 표시

### 관리자 페이지

- `/admin`
  - 관리자 로그인 페이지
  - 로그인 이후 관리자 대시보드 표시

### 향후 추가 예정 유틸 경로

- `/week-calendar`
- `/diff`
- 이후 필요한 유틸을 서브 디렉토리 또는 페이지 형태로 추가

## 5. 홈 페이지 요구사항

- 홈 페이지는 처음에는 비어 있는 상태를 지원해야 한다
- 표시 가능한 유틸이 하나도 없으면 빈 상태 안내 문구를 보여준다
- 예시 문구:
  - `아직 등록된 유틸이 없습니다`
- 우측 상단에는 항상 `/admin`으로 이동할 수 있는 버튼이 있어야 한다
- 필요하면 `/about`으로 이동하는 버튼 또는 링크를 함께 둘 수 있다
- 유틸이 존재하면 리스트 또는 카드 형태로 노출한다
- 각 유틸은 아래 정보를 표시한다
  - 제목
  - 설명
  - 이동 URL

## 6. 관리자 페이지 요구사항

### 6.1 접근 제어

- 관리자 페이지 경로는 `/admin`이다
- 관리자 기능은 로그인 후에만 접근할 수 있다

### 6.2 로그인 정보

- `id`는 고정값 `admin`이다
- `password` 규칙은 아래와 같다
  - `.env`의 `PASSWORD` 값 + 현재 날짜 `MMDD`
- 예시
  - `.env`의 `PASSWORD=abc`
  - 날짜가 5월 3일이면 비밀번호는 `abc0503`

### 6.3 날짜 기준

- 비밀번호 생성에 사용하는 날짜 기준은 `Asia/Seoul`로 고정한다

### 6.4 관리자 기능

- 메모장
  - 관리자 메모를 작성, 수정, 저장할 수 있어야 한다
- 파일 관리
  - 파일 업로드가 가능해야 한다
  - 업로드된 파일 목록을 확인할 수 있어야 한다
  - 업로드된 파일을 다운로드할 수 있어야 한다
- 소개 페이지 관리
  - 내 소개 문구를 수정할 수 있어야 한다
  - 내 사진을 업로드 및 변경할 수 있어야 한다
  - 내 프로젝트 목록을 등록, 수정, 삭제할 수 있어야 한다
  - LinkedIn 등 외부 링크를 등록, 수정, 삭제할 수 있어야 한다
- 유틸 관리
  - 유틸 항목을 직접 등록, 수정, 삭제할 수 있어야 한다
- Visualization 관리
  - 홈에 표시할 유틸 메타데이터를 관리할 수 있어야 한다

## 7. Visualization 정의

Visualization은 홈 화면에 표시할 유틸 정보를 담는 메타데이터 단위로 정의한다.

각 항목은 아래 필드를 가진다.

- `title`
  - 유틸 이름
- `description`
  - 유틸 설명
- `url`
  - 서브 디렉토리 URL
  - 예: `/week-calendar`
- `visible`
  - 홈 화면 노출 여부
- `sortOrder`
  - 홈 화면 정렬 순서

## 8. 데이터 관리 원칙

- 유틸 목록은 파일 시스템을 자동 탐색하지 않는다
- 유틸 목록은 관리자 페이지에서 수동으로 관리한다
- 홈 화면에는 `visible = true`인 항목만 노출한다
- 실제 유틸 페이지는 나중에 별도로 구현할 수 있으며, 메타데이터와 독립적으로 존재할 수 있다
- 소개 페이지 데이터도 관리자 페이지에서 수동으로 관리한다
- 소개 페이지의 사진은 스토리지에 저장하고, 텍스트/링크/프로젝트 정보는 DB에 저장한다

## 10. UI / 디자인 방향

- 전체 UI는 Codex 앱 느낌의 깔끔하고 차분한 스타일을 참고한다
- 과한 블로그 테마보다는 미니멀한 패널형 레이아웃을 우선한다
- 중성적인 컬러와 충분한 여백을 사용한다
- 홈과 관리자 모두 단정하고 관리하기 쉬운 화면 구조를 목표로 한다

## 11. 초기 구현 범위

초기 버전에서는 아래 범위를 우선 구현한다.

- 기본 홈 화면 뼈대
- 우측 상단 관리자 이동 버튼
- 공개 소개 페이지
- 관리자 로그인
- 홈 빈 상태 UI
- 관리자 대시보드 뼈대
- 메모장 기능
- 파일 업로드 / 다운로드 기능
- Visualization CRUD
- 소개 페이지 콘텐츠 CRUD

`/week-calendar`, `/diff` 같은 실제 유틸 구현은 이후 단계에서 추가한다.

## 12. 무료 운영 전제

이 프로젝트는 개인 사용, 낮은 트래픽, 적은 파일 저장량을 기준으로 무료 플랜 안에서 운영 가능한 수준을 목표로 한다.

전제 조건:

- 트래픽이 많지 않다
- 업로드 파일 수와 용량이 크지 않다
- 데이터베이스 사용량이 작다
- 개인용 또는 소규모 운영이다

향후 비용 발생 가능성이 있는 경우:

- 파일 저장 용량이 커지는 경우
- 다운로드 트래픽이 많아지는 경우
- 방문자 수와 요청 수가 크게 증가하는 경우
- 상업적 규모로 운영하는 경우

## 13. 초기 버전 비목표

초기 버전에서는 아래 항목을 우선 범위에서 제외한다.

- 다중 관리자 계정
- 소셜 로그인
- 자동 라우트 탐색
- 고급 통계 기능
- 공개용 파일 관리 화면
- 복잡한 CMS 기능
- 다중 프로필 관리

## 14. 구현 원칙

이 시스템은 처음에는 단순하고 유지보수하기 쉬운 구조로 시작해야 한다.  
이후 유틸이 늘어나더라도 전체 구조를 다시 뜯어고치지 않고 자연스럽게 확장할 수 있어야 한다.
