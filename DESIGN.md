# MyBlogRenewal 설계 문서

## 1. 문서 목적

이 문서는 [REQUIREMENTS.md](D:/Development/MyBlogRenewal/REQUIREMENTS.md)를 바탕으로  
`MyBlogRenewal` 프로젝트의 초기 구현 설계를 정리한 문서이다.

목표는 아래와 같다.

- 초기 버전을 빠르게 구현할 수 있도록 구조를 명확히 한다
- 무료 플랜 범위에서 운영 가능한 방향으로 설계한다
- 이후 유틸이 늘어나더라도 쉽게 확장할 수 있도록 기반을 만든다
- 소개 페이지와 유틸 허브를 함께 운영할 수 있도록 구조를 명확히 한다

## 2. 전체 아키텍처

구성은 크게 3개 영역으로 나눈다.

- `Next.js`
  - 화면 렌더링
  - 관리자 로그인 처리
  - 관리자 페이지 보호
  - 홈 및 각 유틸 페이지 라우팅
  - 공개 소개 페이지 렌더링
- `Supabase Database`
  - 메모 데이터 저장
  - Visualization 메타데이터 저장
  - 업로드 파일 메타데이터 저장
  - 프로필 소개 정보 저장
  - 프로젝트 목록 저장
  - 외부 링크 저장
- `Supabase Storage`
  - 업로드 파일 실제 저장
  - 프로필 사진 저장

## 3. 설계 원칙

- 처음에는 단순하게 시작한다
- 관리자 기능은 모두 서버 검증을 거친다
- 공개 페이지와 관리자 페이지를 명확히 분리한다
- 유틸 목록은 파일 자동 탐색이 아니라 관리자 수동 관리 방식으로 간다
- 파일은 기본적으로 private 접근을 전제로 설계한다
- 확장 전까지는 단일 관리자 기준으로 설계한다

## 4. 라우팅 설계

### 공개 영역

- `/`
  - 홈 페이지
  - 공개 가능한 Visualization 목록 표시
- `/about`
  - 공개 소개 페이지
  - 프로필 사진, 인사말, 소개, 프로젝트 목록, 외부 링크 표시
- `/week-calendar`
  - 향후 유틸 페이지
- `/diff`
  - 향후 유틸 페이지

### 관리자 영역

- `/admin`
  - 로그인하지 않은 경우 로그인 화면 표시
  - 로그인된 경우 관리자 대시보드로 리다이렉트
- `/admin/dashboard`
  - 관리자 대시보드 메인
- `/admin/profile`
  - 소개 페이지 관리 화면
- `/admin/notes`
  - 메모장 관리 화면
- `/admin/files`
  - 파일 업로드 / 다운로드 관리 화면
- `/admin/visualizations`
  - Visualization 목록 / 생성 / 수정 / 삭제 관리 화면

초기 구현에서는 `/admin` 하나의 페이지 안에서 탭 형태로 구성해도 되지만,  
확장성과 유지보수를 위해 내부 경로를 분리하는 방식을 권장한다.

## 5. 인증 및 세션 설계

### 5.1 인증 방식

요구사항에 따라 관리자 인증은 커스텀 로그인으로 구현한다.

- 아이디: `admin`
- 비밀번호: `.env`의 `PASSWORD` 값 + 현재 날짜 `MMDD`

예시:

- `PASSWORD=abc`
- 5월 3일이면 `abc0503`

### 5.2 날짜 계산 기준

- 날짜는 반드시 `Asia/Seoul` 기준으로 계산한다
- 서버에서만 계산한다
- 클라이언트 시간이 아니라 서버 로직을 기준으로 판정한다

### 5.3 인증 처리 방식

권장 방식:

- 로그인 폼 제출
- Next.js 서버 액션 또는 Route Handler에서 인증
- 성공 시 `httpOnly` 쿠키 발급
- 이후 `/admin/*` 접근 시 쿠키 검증

쿠키 저장 정보 예시:

- `admin_session=true`
- 또는 서명된 세션 토큰

초기 버전에서는 간단한 세션 쿠키로 시작할 수 있지만, 아래 조건은 반드시 지킨다.

- 쿠키는 `httpOnly`
- 운영 환경에서는 `secure=true`
- `sameSite=lax` 이상 사용

### 5.4 관리자 페이지 보호

관리자 보호는 UI 숨김이 아니라 서버 기준으로 처리해야 한다.

권장 방법:

- `middleware.ts`에서 `/admin/dashboard`, `/admin/notes`, `/admin/files`, `/admin/visualizations` 접근 제어
- 또는 각 서버 컴포넌트 / Route Handler에서 세션 검증

초기에는 `middleware + 서버 재검증` 조합이 가장 안전하다.

## 6. 홈 페이지 설계

### 6.1 역할

홈 페이지는 블로그 메인이라기보다 유틸 허브 역할을 한다.

### 6.2 데이터 표시 규칙

- `visualizations` 테이블에서 `visible = true`인 항목만 조회
- `sort_order` 오름차순으로 정렬
- 값이 없으면 빈 상태 UI 표시

### 6.3 UI 구성

- 상단 우측: `/admin` 이동 버튼
- 필요 시 `/about` 이동 링크 추가
- 메인 영역:
  - 제목 또는 간단한 소개 문구
  - 유틸 카드 리스트 또는 빈 상태 메시지

## 6A. 소개 페이지 설계

### 6A.1 역할

소개 페이지는 이 사이트의 주인을 보여주는 공개 프로필 페이지이다.

유틸 허브와는 별개로, 방문자가 이 사이트가 누구의 공간인지 자연스럽게 이해할 수 있도록 한다.

### 6A.2 표시 구성

소개 페이지는 아래 순서로 구성한다.

- 프로필 사진
- 간단한 인사말
- 소개 또는 약력
- 그동안 진행한 프로젝트 목록
- LinkedIn 등 외부 링크 목록

### 6A.3 UI 구성

권장 레이아웃:

- 상단
  - 사이트 제목
  - 홈 이동 링크
  - admin 이동 링크
- 메인 히어로 영역
  - 프로필 사진
  - 짧은 인사말
  - 간단한 소개 문장
- 하단 정보 영역
  - 프로젝트 목록 카드 또는 리스트
  - 외부 링크 버튼 또는 리스트

### 6A.4 빈 상태 처리

소개 정보가 비어 있을 가능성도 있으므로 아래 원칙을 둔다.

- 프로필 사진이 없으면 기본 플레이스홀더 사용
- 프로젝트 목록이 비어 있으면 섹션은 유지하되 안내 문구 표시
- 외부 링크가 비어 있으면 해당 영역은 숨기거나 간단히 생략 가능

### 6.4 빈 상태

초기에는 유틸이 없을 가능성이 높으므로, 빈 상태 UI를 정식 상태로 취급한다.

예시:

- `아직 등록된 유틸이 없습니다`

## 7. 관리자 대시보드 설계

### 7.1 공통 구조

관리자 화면은 아래 공통 레이아웃을 갖는다.

- 상단 바
  - 서비스명
  - 로그아웃 버튼
- 좌측 또는 상단 내비게이션
  - 소개 페이지 관리
  - 대시보드
  - 메모장
  - 파일 관리
  - Visualization 관리
- 메인 컨텐츠 영역

### 7.2 대시보드 메인

대시보드 메인에서는 요약 정보만 보여준다.

- 등록된 Visualization 수
- 업로드된 파일 수
- 최근 수정된 메모 또는 최근 업로드 파일

초기 버전에서는 단순 링크 허브 역할만 해도 충분하다.

## 7A. 소개 페이지 관리 설계

### 7A.1 초기 정책

소개 페이지는 `단일 프로필` 기준으로 설계한다.

즉, 여러 사람의 프로필을 관리하는 구조가 아니라  
사이트 주인 1명의 정보를 관리하는 방식으로 구현한다.

### 7A.2 관리 항목

관리자는 아래 내용을 수정할 수 있어야 한다.

- 프로필 사진
- 짧은 인사말
- 소개 / 약력
- 프로젝트 목록
- 외부 링크 목록

### 7A.3 프로젝트 목록 구조

프로젝트 목록은 반복 가능한 목록 데이터로 관리한다.

각 항목 필드 예시:

- `title`
- `description`
- `project_url`
- `sort_order`

### 7A.4 외부 링크 구조

외부 링크도 반복 가능한 목록 데이터로 관리한다.

각 항목 필드 예시:

- `label`
  - 예: `LinkedIn`, `GitHub`, `Portfolio`
- `url`
- `sort_order`

## 8. 메모장 설계

### 8.1 초기 정책

메모장은 초기 버전에서 `단일 메모 문서` 방식으로 구현한다.

이유:

- 구현이 가장 단순하다
- 관리자 1인 사용에 적합하다
- 다중 메모 관리 UI를 만들 필요가 없다

### 8.2 저장 방식

`admin_notes` 테이블에 단일 레코드를 두고 갱신한다.

권장 필드:

- `id`
- `content`
- `created_at`
- `updated_at`

운영 방식:

- 최초 접근 시 레코드가 없으면 생성
- 이후에는 같은 레코드를 계속 수정

## 9. 파일 관리 설계

### 9.1 저장 구조

파일은 `Supabase Storage`에 저장하고, 메타데이터는 DB에 저장한다.

- 실제 파일: Storage bucket
- 파일 정보: `uploaded_files` 테이블

### 9.2 Storage 정책

초기 버전 권장:

- bucket 이름 예시: `admin-files`
- bucket 접근: private

이유:

- 업로드 파일이 공개되면 안 되는 경우를 기본값으로 잡는 것이 안전하다
- 다운로드 시 인증 후 signed URL을 발급하는 방식이 적합하다

### 9.3 업로드 흐름

1. 로그인된 관리자만 업로드 가능
2. 서버에서 파일 검증
3. Supabase Storage에 업로드
4. 성공 시 `uploaded_files`에 메타데이터 저장

### 9.4 다운로드 흐름

1. 로그인된 관리자만 다운로드 요청 가능
2. 서버에서 파일 존재 여부 확인
3. signed URL 또는 서버 경유 응답으로 다운로드 처리

### 9.5 고려할 제한

- 무료 플랜을 고려해 업로드 파일 크기 제한 필요
- 초기 버전에서는 파일 개수와 파일 크기를 보수적으로 제한하는 것이 좋다
- 예: 단일 파일 최대 10MB 또는 더 낮게 제한

## 10. Visualization 설계

### 10.1 개념

Visualization은 홈에 노출할 유틸 항목의 메타데이터이다.

현재 요구사항 기준으로는 `유틸 목록`과 `Visualization`을 분리하지 않고  
`visualizations` 하나로 통합 관리하는 것이 가장 단순하다.

### 10.2 필드 정의

권장 필드:

- `id`
- `title`
- `description`
- `url`
- `visible`
- `sort_order`
- `created_at`
- `updated_at`

### 10.3 데이터 검증 규칙

- `title`
  - 필수
  - 최대 길이 제한 필요
- `description`
  - 선택 또는 필수 여부는 UI 정책에 따라 결정
- `url`
  - `/`로 시작해야 함
  - 중복 방지 필요
- `visible`
  - 기본값 `false` 또는 `true` 중 정책 결정
  - 초기에는 `false` 기본값도 안전함
- `sort_order`
  - 숫자형
  - 기본값 부여 필요

### 10.4 주의점

관리자에서 URL만 먼저 등록하고 실제 페이지가 아직 없을 수 있다.

대응 방법:

- 초기에는 운영자가 직접 관리하는 것으로 전제
- 필요 시 추후 `활성 링크 검사` 기능 추가 가능

- `mime_type` `text`
- `created_at` `timestamp`

### 11.3 `visualizations`

용도:

- 홈 노출용 유틸 메타데이터 저장

권장 컬럼:

- `id` `uuid`
- `title` `text`

- `created_at` `timestamp`
- `updated_at` `timestamp`

### 11.5 `profile_projects`

용도:

- 소개 페이지에 표시할 프로젝트 목록 저장

권장 컬럼:

- `id` `uuid`
- `title` `text`

- `project_url` `text`
- `sort_order` `integer`
- `created_at` `timestamp`

### 11.6 `profile_links`

용도:

- LinkedIn 등 외부 링크 목록 저장

권장 컬럼:

- `id` `uuid`
- `label` `text`
- `url` `text`
- `sort_order` `integer`
- `created_at` `timestamp`

### 11.7 인덱스 / 제약 조건

권장 사항:

- `visualizations.url` 유니크 제약
- `visualizations.sort_order` 인덱스
- `visible` 조회가 많으면 복합 인덱스 고려
- `profile_projects.sort_order` 인덱스
- `profile_links.sort_order` 인덱스

## 12. API / 서버 기능 설계

초기 버전에서는 내부 구현을 아래 방식 중 하나로 구성할 수 있다.

- `Server Actions`
- `Route Handlers`

권장 방향:

- 로그인 / 로그아웃: `Route Handlers`
- 메모 저장: `Server Actions` 또는 `Route Handlers`
- 파일 업로드 / 다운로드: `Route Handlers`
- Visualization CRUD: `Server Actions` 또는 `Route Handlers`
- 프로필 정보 저장: `Server Actions` 또는 `Route Handlers`
- 프로필 사진 업로드: `Route Handlers`

파일 업로드와 인증은 서버 경계가 명확한 `Route Handlers`가 다루기 편하다.

## 13. 환경 변수 설계

필요한 환경 변수 예시:

- `PASSWORD`
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`
- `SUPABASE_SECRET_KEY`

주의사항:

- `PASSWORD`는 절대 클라이언트에 노출되면 안 된다
- `SUPABASE_SECRET_KEY`는 서버 전용으로만 사용한다
- 공개 페이지에서 필요한 값만 `NEXT_PUBLIC_` 접두사를 사용한다

프로필 사진도 스토리지에 저장하므로, 해당 업로드 경로 역시 서버에서만 제어한다.

## 14. 보안 설계

### 14.1 최소 보안 원칙

- 관리자 인증 검증은 서버에서 수행
- 관리자 세션은 `httpOnly` 쿠키 사용
- 파일 버킷은 private 기본값 사용
- 민감한 환경 변수는 서버 전용

### 14.2 추가로 고려할 수 있는 항목

- 로그인 실패 횟수 제한
- 간단한 rate limit
- 관리자 로그 기록

초기 버전에서는 필수는 아니지만, 공개 배포 시 고려 가치가 높다.

## 15. UI 설계 방향

### 15.1 시각적 톤

- Codex 앱 느낌의 정돈된 인터페이스
- 과하지 않은 중성 톤
- 패널, 카드, 얇은 구분선 중심
- 복잡한 장식보다 정보 밀도와 가독성을 우선

### 15.2 홈 화면

- 간단한 제목
- 유틸 목록
- 비어 있으면 empty state
- 우측 상단 admin 버튼
- 필요 시 소개 페이지 이동 링크

### 15.3 소개 페이지

- 프로필 사진이 중심이 되는 조용한 히어로 섹션
- 짧은 인사말과 담백한 소개 문구
- 프로젝트 목록은 너무 화려하지 않은 카드 또는 리스트
- LinkedIn 등 외부 링크는 버튼형 또는 텍스트 링크형
- 전체적으로 개인 취미 공간처럼 보이되 과시적인 포트폴리오 느낌은 줄인다

### 15.4 관리자 화면

- 관리 도구답게 기능 중심 UI
- 폼, 테이블, 리스트 중심
- 모바일에서도 깨지지 않는 반응형 구성

## 16. 무료 플랜 관점 설계 판단

현재 요구사항은 소규모 개인 사이트 기준으로 무료 플랜 운영이 가능한 편이다.

다만 아래는 주의가 필요하다.

- 파일 저장량 증가
- 다운로드 트래픽 증가
- 관리자 기능 확장으로 인한 서버 처리 증가

따라서 초기 버전에서는 아래 정책을 권장한다.

- 파일 크기 제한 도입
- 대용량 미디어 업로드 지양
- 단순한 DB 구조 유지
- 불필요한 실시간 기능 미도입
- 프로필 사진은 고용량 이미지보다 최적화된 이미지 사용

## 17. 구현 우선순위

### 1단계

- Next.js 기본 프로젝트 구성
- 전역 레이아웃 및 기본 테마 구성
- 홈 페이지 뼈대 구현
- `/admin` 진입 버튼 구현

### 2단계

- 관리자 로그인 구현
- 세션 쿠키 처리
- 관리자 페이지 보호 로직 구현

### 3단계

- Supabase 연동
- `visualizations` CRUD 구현
- `profile`, `profile_projects`, `profile_links` CRUD 구현
- 홈과 관리자 데이터 연결

### 4단계

- 메모장 기능 구현
- 파일 업로드 / 다운로드 기능 구현
- 프로필 사진 업로드 및 소개 페이지 연결

### 5단계

- UI 정리
- 예외 처리
- 배포 설정

## 18. 향후 확장 포인트

- 실제 유틸 페이지 추가
- 소개 페이지 테마 고도화
- 관리자 다중 계정 체계
- 유틸 카테고리 분류
- 파일 태그 / 설명 관리
- 최근 업데이트 표시
- 유틸 공개 일정 관리

## 19. 결론

이 프로젝트의 초기 구조는 복잡한 CMS가 아니라  
`관리자가 직접 관리하는 개인용 유틸 허브`로 설계하는 것이 가장 적절하다.

핵심은 아래 3가지이다.

- 홈은 `visualizations` 기반으로 단순하게 운영
- 소개 페이지는 단일 프로필 구조로 가볍게 운영
- 관리자는 서버 검증 기반 커스텀 로그인으로 보호
- 파일과 메모는 최소 기능부터 안정적으로 구현

이 설계를 기준으로 구현하면 초기 범위를 무리 없이 완성하면서도  
이후 유틸 확장에 대응할 수 있는 구조를 만들 수 있다.
